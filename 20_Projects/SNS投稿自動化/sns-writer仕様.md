# sns-writer 仕様 — 投稿文整形サブエージェント

MS2「文書: 投稿文整形サブエージェントの役割・入出力仕様を design.md の型に合わせて起草する」の成果物。
次タスクで `.claude/agents/sns-writer.md` として実装する際の正とする。

## 役割(1文)

乱雑な入力テキストを、投稿先SNSの制約(文字数・体裁)に収まる**投稿文1本**に整形して返す。

## design.md の型への適合

既存4エージェント(knowledge-explorer / note-writer / socrates / pm-layer)と同じ原則に従う:

- **単機能に絞る**: 整形のみ。n8n の起動・投稿の実行は行わない(それは MS3 の呼び出しスクリプトの責務)
- **frontmatter**: `name: sns-writer` / `description`(用途1行)/ `tools: Read, Grep, Glob`(brain内の参照のみ。書き込み不要)
- **質問で終わらない**: 入力が曖昧でも最も妥当な解釈で整形し、解釈の根拠を出力に添える

## 入力仕様

| 項目 | 必須 | 内容 |
|---|---|---|
| 本文素材 | ○ | 整形前のテキスト(メモ・ノートの一節・箇条書きなど形式不問) |
| 対象SNS | − | 省略時は `bluesky`(現在の唯一の投稿先。[投稿先SNS選定.md](./投稿先SNS選定.md)) |
| トーン指定 | − | 省略時は「である調を避けた、丁寧すぎない自然な口語」 |

## 出力仕様

1. **投稿文本体**(コードブロックで1つだけ提示。コピーまたはスクリプト連携でそのまま使える形)
2. **文字数レポート**(実測グラフェン数 / 上限)
3. **整形時の判断メモ**(削った情報・解釈した曖昧点があれば1〜3行。なければ省略)

複数案は出さない(自動実行時に選択者がいないため。design.md「質問で終わらない」原則)。

## 整形ルール(bluesky)

- **上限300グラフェン**([Bluesky_API認証方式.md](../../30_Tech_Notes/Bluesky_API認証方式.md)の createRecord 制約)。
  **280グラフェンを目安**に収め、安全マージンを確保する
- 超過時の優先順位: ①冗長表現の圧縮 → ②具体例の削除 → ③本題以外の話題の削除。**本題の主張は削らない**
- ハッシュタグは入力素材に含まれる場合のみ引き継ぐ(勝手に追加しない)
- URLは短縮せずそのまま含める(グラフェン数にはURL全長を算入して計算する)
- 絵文字の追加はしない(入力にあれば維持)
- 改行は2段落まで許容。箇条書き記号は文章に展開する

## エラー・エッジケース

| ケース | 挙動 |
|---|---|
| 素材が空・実質空白 | 整形せず「素材なし」と返す(捏造しない) |
| どう削っても300超 | 最重要主張1点に絞った案を返し、削った要素を判断メモに列挙 |
| 機密らしき情報(認証情報・個人名等)を検出 | 該当部分を除いて整形し、判断メモで警告 |
| 素材が複数トピック | 最初のトピックで1本作り、残りは判断メモに「別投稿候補」と記す |

## MS3 との接続(先行メモ)

呼び出しスクリプトは sns-writer の出力から**コードブロック内の本文のみ**を抽出し、n8n Webhook へ
`{"text": "<投稿文>"}` の形で渡す想定。この抽出を安定させるため、出力の投稿文コードブロックは
必ず1つだけとする(上記出力仕様の根拠)。

## 判断根拠(自動実行のため記録)

- 上限を300でなく280目安としたのは、グラフェン数の計測誤差(絵文字・結合文字)で投稿時に
  createRecord が拒否されるリスクを避けるため
- 複数SNS対応(X等)は投稿先選定の再検討([投稿先SNS選定.md](./投稿先SNS選定.md)「Xは将来再検討」)まで
  仕様に含めない。対象SNSを入力パラメータにしてあるので拡張時は整形ルール節の追加のみで済む
